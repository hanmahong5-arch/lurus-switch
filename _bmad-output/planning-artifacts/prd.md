## 1. Executive Summary

Lurus Switch 从"AI CLI 配置管理器"转型为**本地 AI 接入平台**。

核心理念：**用户装一个 Switch，所有 AI 工具都能用，充一次钱所有工具共享。**

这不是一个给开发者省事的小工具，而是一个让每个用户获得 360 度价值的平台：

- **下游**：所有主流 AI 工具（CLI/IDE/桌面/脚本）无缝接入
- **中间**：本地网关透明代理 + 智能路由 + 用量计量
- **上游**：Lurus Cloud 统一计费，批量采购 Provider 额度

## 2. Problem Statement

### 2.1 当前痛点矩阵

| # | Pain Point | Severity | 谁痛 | 当前方案 |
|---|-----------|----------|------|---------|
| P1 | 每个 AI 工具单独配 API Key，换 Provider 要改 N 处 | Critical | 所有用户 | 手动一个个改 |
| P2 | 多工具多账号多账单，月底不知道总共花了多少 | High | 付费用户 | 各平台分别查 |
| P3 | 新工具安装后配置繁琐（Key/代理/MCP/模型） | High | 新手 | 读文档试错 |
| P4 | 没有跨工具的智能路由（按任务选模型、自动降级） | Medium | 重度用户 | 手动切换 |
| P5 | 应用开发者要自己处理 LLM 接入的 Key 管理和计费 | High | 开发者 | 自建或让用户 BYO Key |
| P6 | 中国用户的代理/网络配置额外复杂 | High | 中国用户 | Clash/V2Ray 手动配 |

### 2.2 "360 度价值"的含义

Switch 不是解决一个痛点，而是**覆盖用户使用 AI 工具的完整生命周期**：

```
发现 → 安装 → 配置 → 使用 → 付费 → 监控 → 优化 → 升级
  ↑                                                    │
  └────────────────────────────────────────────────────┘
```

每一步都有 Switch 的价值：

| 生命周期阶段 | 没有 Switch | 有 Switch |
|------------|-----------|----------|
| **发现** | 自己搜索工具，不知道哪个好用 | App Store 展示热门 AI 工具 + 评分 |
| **安装** | 每个工具不同安装方式（npm/pip/brew/binary） | 一键安装，Switch 统一管理 |
| **配置** | 手动编辑 JSON/TOML/YAML，配 Key、代理、MCP | 一键接入 Switch 网关，零配置 |
| **使用** | 各工具各自调 API，互不相干 | 所有调用走 Switch，统一路由+缓存 |
| **付费** | 每个平台单独充值/绑卡 | 充一次钱，所有工具通用 |
| **监控** | 不知道哪个工具花了多少钱 | Dashboard 实时展示 per-app/per-model 消费 |
| **优化** | 不知道怎么省钱 | 智能推荐：任务简单时自动用便宜模型 |
| **升级** | 手动检查各工具更新 | 一键全量更新 |

## 3. AI Tool Coverage Map / 全量 AI 工具支持矩阵

### 3.1 工具分类与接入方式

Switch 通过 **4 种接入层级** 覆盖当前市场上所有主流 AI 工具：

#### Tier 1: 全自动托管（Switch 自动检测 + 一键注入配置）

| Tool | Vendor | Config Path | Format | 接入原理 |
|------|--------|------------|--------|---------|
| **Claude Code** | Anthropic | `~/.claude/settings.json` | JSON | 写入 `apiUrl` + `apiKey` 字段 |
| **Codex CLI** | OpenAI | `~/.codex/config.toml` | TOML | 写入 `provider.base_url` + `api_key` |
| **Gemini CLI** | Google | `~/.gemini/settings.json` | JSON | 写入 `apiEndpoint` + `apiKey` |
| **Aider** | Paul Gauthier | `~/.aider.conf.yml` | YAML | 写入 `openai-api-base` + `openai-api-key` |
| **PicoClaw** | Community | `~/.picoclaw/config.json` | JSON | 写入 endpoint + key |
| **NullClaw** | Community | `~/.nullclaw/config.json` | JSON | 写入 endpoint + key |
| **ZeroClaw** | Community | `~/.zeroclaw/config.json` | JSON | 写入 endpoint + key |
| **OpenClaw** | Community | `~/.openclaw/openclaw.json` | JSON | 写入 gateway endpoint + key |

用户操作：**零步** — Switch 自动检测已安装工具，后台静默注入。

#### Tier 2: 半自动引导（Switch 提供指令，用户在工具内粘贴）

| Tool | Vendor | 接入原理 | 用户操作 |
|------|--------|---------|---------|
| **Cursor** | Anysphere | Settings → Models → Override OpenAI Base URL | Switch 生成配置 → 用户粘贴 URL + Key |
| **Windsurf** | Codeium | Settings → Custom Model → API endpoint | 同上 |
| **Trae** | ByteDance | Settings → Model Provider → Custom | 同上 |
| **Zed AI** | Zed Industries | `settings.json` → `"language_models"` → `"api_url"` | Switch 生成 JSON 片段 → 用户粘贴 |
| **Continue** | Continue.dev | `~/.continue/config.json` → `"apiBase"` | Switch 自动写入或提供片段 |
| **Cline** | Saoud Rizwan | VS Code Settings → API Base URL | Switch 提供 URL |

用户操作：**1-2 步** — Switch 显示"复制此内容粘贴到 XXX"的引导卡片。

#### Tier 3: 环境变量注入（设置系统/用户级环境变量）

| Tool/场景 | 环境变量 | 接入原理 |
|----------|---------|---------|
| **Python openai SDK** | `OPENAI_API_BASE`, `OPENAI_API_KEY` | Switch 设置用户环境变量 |
| **LangChain** | `OPENAI_API_BASE`, `OPENAI_API_KEY` | 同上 |
| **CrewAI / AutoGen** | `OPENAI_API_BASE`, `OPENAI_API_KEY` | 同上 |
| **Node.js openai SDK** | `OPENAI_API_BASE`, `OPENAI_API_KEY` | 同上 |
| **任何 OpenAI-compatible 脚本** | `OPENAI_API_BASE` | 同上 |

用户操作：**一键** — Switch 提供"设置全局环境变量"按钮。设置后所有终端新进程自动生效。

#### Tier 4: 通用（任何能设 base_url 的应用）

任何支持 OpenAI-compatible API 的应用，只需指向：
```
Base URL: http://localhost:19090
API Key:  sk-switch-xxxx (Switch 生成的 app token)
```

用户操作：**手动输入 URL + Key**，Switch "应用管理"页面提供一键复制。

### 3.2 接入覆盖率目标

| 时间节点 | Tier 1 工具数 | Tier 2 工具数 | 用户操作上限 |
|---------|-------------|-------------|------------|
| Phase 1 (Month 2) | 8 (现有) | 0 | Tier 1 零步 |
| Phase 2 (Month 4) | 8 | 6 (Cursor/Windsurf/Trae/Zed/Continue/Cline) | Tier 2 一步 |
| Phase 3 (Month 6) | 10+ | 10+ | Tier 3 环境变量一键 |
| Phase 4 (Month 8+) | 动态扩展 | 社区贡献 | 全覆盖 |

## 4. Target Users

### 4.1 User Persona: Power Coder（核心）

- **画像**: 全栈开发者，每天用 Claude Code + Cursor/Codex，偶尔用 Aider
- **月 AI 消费**: ¥300-2000
- **核心 Job**: 写代码时 AI 工具"能用就行"，不想花时间折腾配置和账单
- **Switch 时刻**: "装上 Switch，配好一次，以后所有工具都不用管了"
- **关键指标**: 首次使用到所有工具通网 < 5 分钟

### 4.2 User Persona: Team Lead（高价值）

- **画像**: 管 5-20 人开发团队，需要统一 AI 工具环境和控制预算
- **月团队 AI 消费**: ¥5000-20000
- **核心 Job**: 标准化团队环境 + 控制费用 + 追踪 ROI
- **Switch 时刻**: "给团队每人装 Switch，统一配置一键下发，费用汇总一目了然"
- **关键指标**: 团队人均 AI 工具配置时间从 2 小时 → 5 分钟

### 4.3 User Persona: Curious Explorer（增长）

- **画像**: 听说 AI 编程很牛，想试试但不知道从哪开始
- **月 AI 消费**: ¥0-50
- **核心 Job**: 最低门槛开始用 AI 编程工具
- **Switch 时刻**: "装 Switch → 充 10 块钱 → 一键装上 Claude Code → 直接用"
- **关键指标**: 从安装 Switch 到发出第一条 AI 请求 < 3 分钟

### 4.4 User Persona: App Developer（供给侧）

- **画像**: 在做一个需要 LLM 的桌面应用/脚本/工具
- **核心 Job**: 不想自己搭计费系统，希望用户已有付费能力
- **Switch 时刻**: "我的应用只要 `base_url = localhost:19090`，用户装了 Switch 就能直接用"
- **关键指标**: 接入文档阅读 → 应用调通 API < 30 分钟

## 6. Functional Requirements

### 6.1 Gateway Core / 网关核心（P0）

| ID | Requirement | Details |
|----|------------|---------|
| G-01 | **OpenAI-compatible API** | 完整实现 `/v1/chat/completions`, `/v1/completions`, `/v1/embeddings`, `/v1/models` |
| G-02 | **Streaming support** | SSE (Server-Sent Events) 流式传输，零额外延迟 |
| G-03 | **Per-app authentication** | 每个注册应用分配独立 `sk-switch-xxx` token，网关验证 |
| G-04 | **Transparent proxy** | 请求透传到 Lurus Cloud NewAPI，不修改请求体 |
| G-05 | **Always-on** | 开机自启 + 系统托盘常驻，网关始终可用 |
| G-06 | **Auto-retry with fallback** | 上游 5xx → 自动重试（指数退避）→ 切备用 Provider |
| G-07 | **Response caching** | 可选，TTL-based LRU 缓存，相同请求复用（非流式） |
| G-08 | **Health endpoint** | `GET /health` 返回网关状态 + 上游连通性 |
| G-09 | **Local control API** | `GET /switch/v1/status`, `/balance`, `/app-token`, `/models` 供第三方应用查询 |
| G-10 | **Graceful degradation** | Lurus Cloud 不可达时，自带 Key 模式的请求仍可直连上游 |

### 6.2 App Registry / 应用注册（P0）

| ID | Requirement | Details |
|----|------------|---------|
| A-01 | **Auto-detect installed tools** | 启动时+定期扫描 PATH、已知配置路径、环境变量 |
| A-02 | **Manual app registration** | 用户手动注册应用：名称、图标 → 获得 token |
| A-03 | **Per-app token management** | CRUD token，支持重置（安全考虑） |
| A-04 | **Cloud app directory** | 从 Lurus Cloud 拉取推荐应用列表 + 接入指南 |
| A-05 | **One-click connect (Tier 1)** | 自动检测到的 Tier 1 工具一键写入网关配置 |
| A-06 | **Guided connect (Tier 2)** | Tier 2 工具提供图文引导（含截图 + 一键复制） |
| A-07 | **Env var injection (Tier 3)** | 一键设置 `OPENAI_API_BASE` 等系统环境变量 |
| A-08 | **New tool detection** | 后台定期扫描，发现新工具弹出桌面通知 |

### 6.3 Metering / 用量计量（P0）

| ID | Requirement | Details |
|----|------------|---------|
| M-01 | **Per-request logging** | 每次 API 调用记录: app_id, model, tokens_in, tokens_out, latency, cost, timestamp |
| M-02 | **Local SQLite storage** | 本地存储，支持 90 天历史，自动归档/清理 |
| M-03 | **Real-time aggregation** | 今日/本周/本月的 per-app, per-model 汇总 |
| M-04 | **Cache hit tracking** | 记录缓存命中次数和节省金额 |
| M-05 | **Export** | 支持导出 CSV/JSON 用量报表 |

### 6.4 Billing / 计费（P0）

| ID | Requirement | Details |
|----|------------|---------|
| B-01 | **Balance display** | 实时余额（从 Lurus Cloud 同步） |
| B-02 | **Top-up** | 支付宝/微信扫码充值 |
| B-03 | **Redeem code** | 兑换码充值 |

## 7. Information Architecture / 信息架构

### 7.1 Navigation

```
Sidebar:
├── 控制台        Dashboard — 网关状态 + 余额 + 今日概览 + 实时活动
├── 已连接        Connected Apps — 工具卡片列表 + 未连接提示
├── 应用商店      App Store — 工具发现 + 安装 + 接入指南
├── 费用         Billing — 余额/充值/趋势图/分维度消费/省钱洞察
├── ────
├── MCP 管理      MCP server 配置 + 预设 + 健康检查
├── Prompt 库     Prompt 模板 + 自定义管理
├── CLAUDE.md     扫描/编辑/评分/模板
├── ────
├── 推广中心      推广链接/返利/统计
└── 设置         网关/预算/缓存/外观/网络/账号
```

### 7.2 核心信息展示优先级

| 位置 | 始终可见的信息 | 意义 |
|------|-------------|------|
| **托盘图标** | 网关状态（颜色）| 一眼知道"系统正常" |
| **托盘 Tooltip** | 余额 + 今日消费 | 不开窗口就知道花了多少 |
| **Dashboard 顶部** | 网关状态 + 余额 + 连接数 | 进入就看到全局状态 |
| **Dashboard 核心** | 今日调用量 + 今日消费 + 缓存节省 | 核心价值量化 |
| **Dashboard 排行** | per-app + per-model 消费 | 钱花哪了一目了然 |
| **Dashboard 活动流** | 最近 API 调用列表 | "Switch 在工作"的信心 |

### 7.3 信息密度设计原则

1. **控制台**: 信息密度高 — 这是用户最常看的页面，要在一屏内展示核心指标
2. **已连接**: 信息密度中 — 每个工具一张卡片，兼顾状态和操作
3. **费用**: 信息密度高 — 图表 + 排行 + 洞察，让用户尽可能长时间停留
4. **应用商店**: 信息密度中低 — 浏览发现为主，不要信息过载
5. **设置**: 信息密度低 — 低频操作，清晰分组即可

## 8. Non-Functional Requirements

### 8.1 Performance

| Metric | Target | Rationale |
|--------|--------|-----------|
| 应用启动 | < 2s | 不能让用户等 |
| 网关就绪 | < 1s (启动后) | 应用随时可能调 API |
| 网关额外延迟 | < 5ms (p99) | 用户不应感知到 Switch 的存在 |
| 内存占用（后台） | < 50MB | 不能干扰用户主力工作 |
| 二进制体积 | < 25MB | 下载要快 |
| 工具扫描 | < 3s (并行) | 首次启动体验 |
| SQLite 写入 | < 1ms/条 | 不能成为网关瓶颈 |

### 8.2 Reliability

| Requirement | Details |
|-------------|---------|
| 网关可用性 | 99.9%（仅统计非维护时段） |
| 配置安全 | 写入工具配置前自动快照 |
| 数据持久 | SQLite + WAL 模式，异常退出不丢数据 |
| 网络断开 | 自带 Key 请求仍可用；Lurus 请求排队或报错 |
| 崩溃恢复 | 网关 panic 自动重启（supervisor 模式） |

### 8.3 Security

| Requirement | Details |
|-------------|---------|
| Token 存储 | OS keychain 优先（Windows Credential Manager / macOS Keychain / GNOME Keyring），fallback AES-256 加密文件 |
| 日志脱敏 | API Key / Token 在日志中自动遮蔽 |
| 本地网关 | 仅监听 `127.0.0.1`，不暴露到网络 |
| 配置导出 | 自动脱敏 API Key（只保留前后 4 位） |
| HTTPS 上游 | 与 Lurus Cloud 通信强制 TLS 1.2+ |

### 8.4 Compatibility

| Platform | Support Level |
|----------|-------------|
| Windows 10/11 (x64) | Tier 1 — 完整功能 |
| macOS 13+ (x64, arm64) | Tier 1 — 完整功能 |
| Linux (x64, Wayland/X11) | Tier 2 — 核心功能，托盘可能需要特殊处理 |

## 11. Technical Architecture Summary

> 完整架构详见 `product-brief.md` Section 5

核心决策:

| Decision | Choice | Rationale |
|----------|--------|-----------|
| 网关实现 | 自建 Go HTTP Server | 轻量 (<30MB 内存)、可控、原生 per-app metering |
| 运行模式 | 系统托盘常驻 | 网关必须 always-on |
| 计量存储 | 嵌入式 SQLite | 零外部依赖、性能足够、90 天滚动 |
| 上游通信 | HTTPS → Lurus Cloud NewAPI | 重活（Provider 管理/计费/路由）在云端 |
| API 兼容 | 严格 OpenAI-compatible | 任何用 OpenAI SDK 的应用零改造 |

## 12. Risks & Mitigations

| Risk | Impact | Probability | Mitigation |
|------|--------|------------|------------|
| 用户不信任本地代理 | 高 | 中 | 开源核心网关代码；安全审计；强调"只监听 localhost" |
| 网关引入感知延迟 | 高 | 低 | 性能基准测试 < 5ms；纯内存路由；无中间件开销 |
| 工具配置格式频繁变更 | 中 | 高 | 宽松解析（保留未知字段）；写入前快照；CI 监控上游 Release |
| 冷启动问题（无用户无应用） | 高 | 高 | 自有 Tier 1 工具作为种子；体验额度降低首次付费门槛 |
| 上游 Provider ToS 风险 | 高 | 低 | 法务审查；定位为"增值服务"而非纯转售 |
| macOS/Linux 系统托盘兼容 | 中 | 中 | `getlantern/systray` 库；Linux fallback 到 AppIndicator |
